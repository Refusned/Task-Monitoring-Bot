# DESIGN.md — Exchange Order Monitor Bot

Сопроводительный документ к тестовому заданию. Описывает архитектуру, решения и
принятые допущения по неполному ТЗ. Дополнение к [CLAUDE.md](../CLAUDE.md).

> **Статус документа:** Day 1 — каркас, расширяется по мере реализации.
> Финальный вид — на Дне 7. Источник истины по решениям — этот файл; CLAUDE.md
> остаётся живым контекст-файлом для будущих сессий.

## Что и зачем

Бот ведёт жизненный цикл заказов на 5 биржах накрутки/микрозадач: создаёт →
мониторит → проверяет результат → принимает оплату или возвращает на доработку.
Покрывает два сценария MVP из ТЗ: накрутка активности (подписки/лайки на новые
посты) и закупка трафика из соцсетей (с проверкой через Яндекс.Метрику и
еженедельной отчётностью в Google Sheets).

ТЗ короткое и неформальное. Чтобы продемонстрировать самостоятельность,
заказчик не запрашивался; все неоднозначности закрыты осознанными допущениями
(раздел «Допущения и решения» ниже), каждое — config-driven (CLAUDE.md → A1-A12).

## Архитектура (кратко)

Подробности — в [CLAUDE.md → Архитектура](../CLAUDE.md#архитектура).

- **Адаптеры — два типа.** `PanelAdapter` (SMM-панели: smmcode, prskill) и
  `TaskExchangeAdapter` (биржи микрозадач: unu, advego, ipgold). Базовый
  `ExchangeAdapter` тонкий: `get_balance` и `capabilities`. Per-операционные
  capability-флаги задают, что биржа умеет; поведение оркестратора при
  отсутствии операции определено.
- **Оркестратор** — конечный автомат поверх адаптеров. Идентичность заказа —
  внутренний `client_order_uuid`. Два набора статусов: `OrderStatus` и
  `SubmissionStatus` (для бирж микрозадач).
- **Проверка результата** — независимая, не доверяем статусу биржи (A4). Для
  трафика — Яндекс.Метрика по UTM (A10). Для активности — ручной baseline +
  финальный замер с программным чтением счётчиков там, где есть API (A3).
- **Telegram-бот** — лицо продукта; создание заказов и подтверждение оплаты.
- **CLI** — движок, тестируемый интерфейс, демо.
- **Хранилище** — SQLite в WAL, единственный источник истины.

## Идемпотентность и деньги (C1 / C2)

- **C1 — не разместить заказ дважды.** Строка `orders` в статусе `CREATING`
  пишется ДО вызова биржи; после успеха — `external_order_id` + `ACTIVE`; при
  старте бота — реконсиляция `CREATING`-строк с биржей до повторного
  `create_order`. Идентичность — `client_order_uuid`, не натуральный хэш.
- **C2 — не оплатить дважды.** `payments` с
  `UNIQUE(exchange, external_submission_id)`; перед вызовом биржи — условный
  апдейт статуса сабмишена; `rowcount = 0` => уже обработано. Транзакция БД не
  держится на время HTTP-вызова: паттерн «занять → commit → внешний вызов →
  записать результат» через `action_log`.

## Допущения и решения (по неполному ТЗ)

Полный реестр — [CLAUDE.md → Принятые допущения](../CLAUDE.md#принятые-допущения-решения-по-неоднозначностям-тз).
Главное:

| # | ТЗ | Решение | Почему |
|---|---|---|---|
| A1 | «мониторить уже размещённые» + «Создание заказов» | бот создаёт и мониторит полный цикл | «Создание заказов» прямо в обоих пунктах MVP |
| A3 | «лайки на новые посты» | бот сам отслеживает новые посты | «новые» = поток во времени; иначе не автоматизация |
| A5 | «оплачивать» | подготовка ботом, финальный «принять» — человек; порог авто — конфиг | реальные деньги, необратимо => senior-дефолт |
| A7 | «уже РАЗМЕЩЕННЫЕ» (заглавными) | импорт заказа по `external_order_id` | акцент в ТЗ |
| A9 | вкладка «Трафик из соц сетей» | свои логичные колонки + адаптация к существующим заголовкам | колонки выводимы из ТЗ; адаптация страхует от расхождения формата |
| A10 | проверка переходов в Метрике | конфиг + мок без доступа | drop-in после получения токена |

«Принципиальная страховка»: всё предполагаемое — параметр конфигурации;
неверная трактовка = правка `.env`/конфига, не переделка кода.

## Где уместен LLM и почему его нет в MVP

Решено: LLM в продукт не закладывается. Обоснование:

- Проверка результата на MVP — числовая: счётчики подписок/лайков, Метрика по
  UTM. Решение «принять/вернуть» — пороговое. LLM здесь только усложнит и
  удорожит.
- Естественные места для LLM — за пределами MVP: качественная проверка
  контентных сабмишенов (тексты комментариев на advego — «по теме / спам /
  валидно»); LLM-агент как страховка для бирж без API. На неделю и без
  подтверждения о роли LLM в продукте — не закладываем. Архитектурно:
  верификатор — отдельный модуль с интерфейсом «вердикт + сырые
  доказательства», LLM-вариант имплементируется drop-in.

## Ограничения и комплаенс

- Работаем только с авторизованными аккаунтами, реквизиты которых даны в
  `test_task.txt`; секреты — в `.env`, не коммитятся, не логируются, не
  показываются в Telegram.
- Лимиты трат — на заказ и суточный, в конфиге, проверяются транзакционно.
- Размещение заказа и приём оплаты — после подтверждения админом (по умолчанию).
- Аудит всех денежных действий: кто/когда/что приняли/вернули, какие метрики
  проверялись, сырые ответы бирж.
- Без обхода анти-бот систем и без скрапинга чужих учётных данных.
