# CLAUDE.md — Exchange Order Monitor Bot

Контекст проекта для Claude Code. Проект ещё не реализован — это начальный файл со
всеми данными. План реализации дважды отревьюен через `/autoplan` + финальная критика
на соответствие ТЗ; все неоднозначности ТЗ закрыты осознанными допущениями (см. раздел
«Принятые допущения»).

## Что это

Бот, управляющий жизненным циклом заказов на 5 биржах накрутки/микрозадач: **создаёт
заказы → мониторит → проверяет результат → при результате оплачивает (принимает),
иначе возвращает на доработку.** Тестовое задание на вакансию ИИ-инженера.

## Тестовое задание (источник истины)

Текст задания — `../test_task.txt` (в родительской папке `New bot/`, вне репозитория).
Дословная суть:

- Бот мониторит **уже размещённые** задания на биржах, проверяет результат: есть
  результат — оплачивает, нет — возвращает на доработку.
- MVP — два сценария:
  1. **Накрутка активности.** Создание заказов на: подписки в наши соцсети, накрутку
     лайков **на новые посты**; проверка исполнения.
  2. **Переходы из соцсетей на сайт.** Создание заказов на переходы, проверка
     исполнения, еженедельная фиксация в таблице. Переходы заказываются из 6
     платформ: **ВК, X, YouTube, Telegram, Дзен, Pinterest**. Количество переходов
     отслеживается в **Яндекс.Метрике**. Отчётная таблица по неделям — Google Sheets,
     вкладка **«Трафик из соц сетей»**.

Срок — ~1 неделя (задание получено 2026-05-22). Деливерабл: рабочий MVP + `docs/DESIGN.md`.

## Стратегия по неполному ТЗ

ТЗ короткое и неформальное. Решено **не запрашивать заказчика**, а закрыть все
неоднозначности самостоятельными осознанными допущениями — продемонстрировать
способность работать без чёткого ТЗ. Все допущения логичны, обоснованы и
**config-driven**: неверная трактовка = правка конфига, не переделка. Полный реестр —
ниже; перейдёт в `DESIGN.md` разделом «Допущения и решения».

## Зафиксированные решения

- **Все 5 бирж** — подтверждено пользователем против рекомендации ревью.
- **Без LLM в продукте** — подтверждено против рекомендации ревью; обоснование «почему
  LLM не в MVP» — отдельный раздел `DESIGN.md`. «ИИ-инженер» здесь = разработка через
  ИИ-инструменты, а не LLM внутри продукта.
- Интерфейс: **Telegram-бот** (лицо продукта) **+ CLI** (движок, тесты, скрипты).
- Деньги: размещение заказа и приём оплаты — после подтверждения админом (по умолчанию).

## Принятые допущения (решения по неоднозначностям ТЗ)

Каждое допущение: что в ТЗ → решение → обоснование → как конфигурируется.

**A1. Бот создаёт И мониторит (не только мониторит).**
ТЗ: строка 1 — «мониторить уже размещённые»; п.1 и п.2 — «Создание заказов на бирже».
Решение: полный цикл — создание, мониторинг, проверка, оплата/возврат.
Обоснование: «Создание заказов» написано прямо в обоих пунктах MVP. «Уже размещённые» —
акцент на ведении заказа по жизненному циклу после размещения. Конфиг: n/a (прямое требование).

**A2. Целевые аккаунты соцсетей и URL сайта — параметры конфигурации.**
ТЗ: «подписки в наши соцсети», «переходы на сайт» — конкретные ресурсы не даны.
Решение: целевые ресурсы задаются в конфиге (`targets`), бот к ним не привязан.
Обоснование: хардкод целевых ресурсов — антипаттерн; config-driven корректен в любом
случае. Конфиг: `targets` (аккаунты, URL сайта) — полностью.

**A3. «Лайки на новые посты» — бот сам отслеживает новые посты.**
ТЗ: «накрутку лайков на новые посты».
Решение: модуль `posts/watcher.py` периодически проверяет целевые аккаунты; новый
пост → инициирует заказ лайков (с подтверждением по политике оплаты).
Обоснование: «новые» = поток постов во времени; бот, требующий ручного указания
каждого поста, не автоматизирует задачу. Конфиг: интервал опроса, вкл/выкл авто-режим,
лимит лайков на пост; есть и ручной режим (указать пост явно).

**A4. «Проверка результата» — независимая, не доверяем статусу биржи.**
ТЗ: «проверять результат».
Решение: бот проверяет фактический результат сам — Метрика для трафика, дельта
счётчиков для активности — не полагаясь на поле «выполнено» от биржи.
Обоснование: смысл бота — перепроверить перед оплатой; доверие бирже сделало бы
проверку бессмысленной. Конфиг: пороги прохождения, окна и повторы проверки.

**A5. «Оплачивать» — подготовка ботом, финальное «принять» подтверждает человек.**
ТЗ: «если результат есть — оплачивать».
Решение: при прохождении проверки бот готовит оплату; приём по умолчанию подтверждает
админ в Telegram. Заложен конфиг-порог авто-приёма для очевидных дешёвых случаев.
Обоснование: «бот оплачивает» — да, но это реальные деньги на чужих аккаунтах,
необратимо; senior-дефолт — человек в контуре, автоматизация управляемо включается.
Конфиг: порог авто-приёма (сумма + уверенность проверки); 0 = всё через человека.

**A6. «Возврат на доработку» — автоматически (где поддерживается).**
ТЗ: «если нет — возвращать на доработку».
Решение: не прошедший проверку сабмишен бот возвращает (reject с причиной)
автоматически — на биржах микрозадач. На SMM-панелях цикла нет (заказ предоплачен).
Обоснование: возврат — не трата денег и обратимая операция, безопасно автоматизировать.
Конфиг: можно перевести возврат тоже на подтверждение человеком.

**A7. «Уже размещённые» — поддержка импорта заказов, созданных вручную.**
ТЗ: заглавными — «мониторить уже РАЗМЕЩЕННЫЕ задания».
Решение: команда импорта заказа по `external_order_id` — бот ведёт и заказы,
размещённые на бирже вручную.
Обоснование: акцент заглавными; полная трактовка — мониторинг любых размещённых
заказов, не только своих. Конфиг: команда импорта.

**A8. Шесть платформ-источников — поддержка всех; покрытие уточняется из ЛК.**
ТЗ: «переходы из ВК, Х, Ютуб, Телеграм, Дзен, Пинтерест».
Решение: поле `source_platform` со всеми 6; матрица «биржа × платформа» строится на
Дне 2 из кабинетов бирж; непокрытое честно отмечается.
Обоснование: покрытие платформ биржами заранее не известно; строим под все 6.
Конфиг: матрица «биржа × платформа» — данные, не код.

**A9. Отчётная таблица — логичная структура + адаптация к существующей.**
ТЗ: «Отчётная таблица по неделям (вкладка Трафик из соц сетей)».
Решение: запись во вкладку «Трафик из соц сетей»; колонки — Неделя, Платформа-источник,
Биржа, Заказано переходов, Фактически (Метрика), Стоимость, Статус. Если у заказчика
уже есть таблица с заголовками — бот считывает заголовки и пишет под них.
Обоснование: колонки выведены из ТЗ (отчёт по неделям × по источникам) + сверочные
поля заказано/фактически + бюджет; чтение существующих заголовков страхует от
расхождения формата. Конфиг: ID таблицы, маппинг колонок.

**A10. Яндекс.Метрика — подключается из конфига; без доступа — мок-режим.**
ТЗ: «количество переходов отслеживается в Яндекс Метрике».
Решение: счётчик + токен в конфиге; `TrafficVerifier` использует Stat API с фильтром
по UTM; без реального доступа — режим мок-верификации (разработка/демо).
Обоснование: внешний доступ может прийти не сразу — нельзя ставить проект в
зависимость от него; реальная Метрика — drop-in. Конфиг: счётчик/токен; мок ↔ реальный.

**A11. Создание заказов доступно и в Telegram, и в CLI.**
ТЗ: «Бот, который...».
Решение: `/new_order` — минимальный флоу в Telegram; `create-order` — команда CLI.
Обоснование: продукт назван «Бот» → создание должно жить в Telegram; CLI нужен для
тестируемости и демо. Конфиг: n/a.

**A12. Формат сдачи — рабочий MVP + DESIGN.md.**
ТЗ: формат не указан.
Решение: рабочий запускаемый бот + `DESIGN.md`.
Обоснование: для теста на инженера рабочий артефакт + документ с обоснованиями —
сильнейшая подача; перекрывает вариант «только описание». Конфиг: n/a.

## Биржи

| Биржа | Тип | API | Цикл accept/rework |
|---|---|---|---|
| smmcode.shop | SMM-панель | reseller API, `api_token` | нет — заказ предоплачен |
| prskill.ru | SMM-панель | reseller API, `key`+`action` | нет — заказ предоплачен |
| unu.im | биржа микрозадач | API v1/v2, `api_key` | да — approve/reject reports |
| advego.com | биржа микрозадач | XML-RPC, `token` | да — принять/вернуть/отклонить |
| ipgold.ru | биржа микрозадач | API заявлен, методы не подтверждены | да (ожидается) |

**Два типа площадок моделируются явно.** SMM-панель: заказ предоплачен при создании,
«проверка» = сверка внешнего результата, цикла accept/rework нет. Биржа микрозадач:
исполнители сдают отчёты-сабмишены, заказчик принимает (оплата) или возвращает каждый.

## Стек

Python 3.11+ · asyncio · `aiogram` 3 (Telegram) · `httpx` (HTTP; advego — XML-RPC
поверх httpx) · `APScheduler` (поллинг, отчёт) · `aiosqlite` в режиме WAL (состояние,
источник истины) · `pydantic` + `pydantic-settings` (конфиг, валидация) · `gspread`
(Google Sheets) · `pytest` + `ruff`.

## Архитектура

```
 CLI (cli.py)                    Telegram bot (aiogram, whitelist админов)
  create-order / run / demo       /new_order /orders /order ; инлайн «Принять/Доработка»
        └──────────────┬────────────────────┘
                       ▼
 Orchestrator ── конечный автомат, идемпотентность, аудит-лог
   ├─▶ Adapters     ExchangeAdapter (тонкий) → PanelAdapter / TaskExchangeAdapter
   │                 per-операционные capability-флаги
   ├─▶ PostWatcher  детектор новых постов целевых аккаунтов (A3)
   ├─▶ Verification TrafficVerifier (Метрика+UTM) / ActivityVerifier
   │                 вердикт: auto_pass / needs_human_review / fail
   └─▶ DB (aiosqlite WAL) — источник истины:
        orders · submissions · payments(UNIQUE) · action_log ·
        verifications · audit_log · report_rows · migrations
   ▲
 Scheduler (APScheduler; max_instances=1, coalesce=True)
   poll_active_orders (per-exchange lock+джиттер) · poll_new_posts · weekly_report
        ▼
 Reporting ─▶ Google Sheets (вкладка «Трафик из соц сетей», только выгрузка)
```

- **Адаптеры:** тонкая база `ExchangeAdapter` (`get_balance`, `capabilities`);
  `PanelAdapter` (`create_order`, `get_order_status`); `TaskExchangeAdapter`
  (`create_campaign`, `list_submissions`, `accept_submission`, `reject_submission`,
  `get_order_status`). Поведение оркестратора при отсутствующей capability определено.
- **Статусы:** `OrderStatus` (DRAFT/CREATING/ACTIVE/VERIFYING/COMPLETED/FAILED/CANCELLED)
  и `SubmissionStatus` (NEW/VERIFYING/AWAITING_ADMIN/ACCEPTING/ACCEPTED/REJECTING/
  REWORK_REQUESTED/FAILED). Идентичность заказа — внутренний `client_order_uuid`.

## Идемпотентность и деньги (критично)

- **C1 — не разместить заказ дважды:** строка `orders` в `CREATING` пишется ДО вызова
  биржи; после успеха — `external_order_id` + `ACTIVE`; при старте — реконсиляция строк
  `CREATING` с биржей до любого повторного `create_order`.
- **C2 — не оплатить дважды:** `payments` с `UNIQUE(exchange, external_submission_id)`;
  перед вызовом биржи — условный апдейт статуса (`rowcount=0` → уже обработано);
  транзакция БД НЕ держится на время HTTP-вызова (паттерн «занять → commit → вызов →
  записать результат» через `action_log`); инлайн-кнопка гасится при нажатии.
- Лимиты трат (на заказ + суточный) проверяются в транзакции вставки заказа; числовые
  поля из ответов бирж валидируются до попадания в логику трат.

## Безопасность — ВАЖНО

- Реквизиты 5 бирж даны в `../test_task.txt` (вне репозитория). При реализации: все
  секреты (логины/пароли/токены/cookie) → `.env`; создать `.gitignore` с `.env` и
  записями секретов; **никогда не коммитить, не логировать, не показывать в Telegram**.
- `.env.example` — шаблон без значений, его коммитить можно.
- `DRY_RUN` — режим без реальных вызовов create/accept (для разработки и демо).
- `DESIGN.md` раздел «Ограничения и комплаенс»: только предоставленные авторизованные
  аккаунты, лимиты трат, аудит денежных действий, без обхода анти-бот систем.
